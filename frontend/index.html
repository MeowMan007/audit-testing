Understanding WCAG Accessibility

The Web Content Accessibility Guidelines (WCAG) 2.1 define how to make web content more accessible. Our tool checks for these key principles:

🎨

Color Contrast

Text must have sufficient contrast against its background. WCAG requires at least 4.5:1 for normal text and 3:1 for large text (18px+). Low contrast makes content unreadable for users with low vision.

WCAG 1.4.3 — Level AA
🖼️

Alt Text

Every informational image must have a descriptive alt attribute. Screen readers read this text aloud to users who cannot see the image. Decorative images should use alt="".

WCAG 1.1.1 — Level A
📐

Semantic HTML

Proper use of HTML elements (<nav>, <main>, <h1>-<h6>) creates a meaningful document structure that assistive technologies rely on for navigation.

WCAG 1.3.1 — Level A
📊

Accessibility Score

Our scoring system starts at 100 and deducts points for each violation found. Critical issues deduct more than warnings. The CNN model adds AI-detected deductions.

Grades: A (90+) → F (below 60)

Known Limitations

🤖

AI Model Limitations

The ViT-B/16 model is trained on a robust real-world dataset of ~5,500 samples (trained in April 2026). It covers approximately 70% of WCAG guidelines, significantly reducing false positives and effectively handling real-world edge cases.

📋

Rule-Based Coverage

Automated rule checks, keyboard navigation simulation, and AI visual analysis cover approximately 75% of WCAG 2.1 criteria. Remaining accessibility issues (like cognitive load assessment or full screen reader compatibility) require manual expert testing.

⚡

Dynamic Content

Single-page applications (SPAs), lazy-loaded content, and JavaScript-heavy sites may not be fully captured. The tool takes a snapshot at load time and may miss content that appears after user interaction.
